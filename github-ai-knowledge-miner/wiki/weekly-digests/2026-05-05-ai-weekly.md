## Top Changes / 本周重点变化

- [LangChain](../repos/langchain-2026-05-week1.md) tightened its core tool model by collapsing schema state into a single `ToolSchema` cache, which is the kind of internal cleanup that usually pays off in fewer invalidation bugs. / [LangChain](../repos/langchain-2026-05-week1.md) 把 schema 状态收敛到单一 `ToolSchema` 缓存里，这类内部整理通常会少掉很多失效和同步问题。
- The same repo also added a `respond` HITL decision, which makes human-in-the-loop flows feel more like a first-class execution path instead of a bolt-on exception case. / 同一个仓库还加入了 `respond` 这个 HITL 决策，让人类介入流程更像一等公民，而不是补丁式分支。
- [vLLM](../repos/vllm-2026-05-week1.md) finished end-to-end NVFP4 KV cache support through FlashInfer, which is a real serving-side capability jump, not a cosmetic kernel tweak. / [vLLM](../repos/vllm-2026-05-week1.md) 通过 FlashInfer 完成了端到端 NVFP4 KV cache 支持，这是实打实的推理服务能力提升，不是表层 kernel 调整。
- [Phoenix](../repos/phoenix-2026-05-week1.md) moved PXI prompt assembly server-side and added prompt caching, which reduces frontend leakage of static instructions and creates a cleaner cache boundary. / [Phoenix](../repos/phoenix-2026-05-week1.md) 把 PXI prompt 组装移到服务端并加入 prompt 缓存，减少了前端对静态指令的暴露，也让缓存边界更清楚。
- [LlamaIndex](../repos/llama-index-2026-05-week1.md) fixed OTel span context attachment, which is the sort of tracing bug that quietly ruins downstream observability if left alone. / [LlamaIndex](../repos/llama-index-2026-05-week1.md) 修复了 OTel span 上下文附着问题，这类 tracing bug 如果不管，通常会悄悄把下游可观测性弄坏。

## New Papers Worth Reading / 新论文值得读

- [RunAgent](../papers/runagent-constraint-guided-execution.md) is the most practically interesting paper here: it treats plan execution as a constrained, stepwise system rather than a free-form chat loop. / [RunAgent](../papers/runagent-constraint-guided-execution.md) 是这里最有实践价值的一篇：它把计划执行当成受约束的逐步系统，而不是自由聊天循环。
- The [procedural-execution diagnostic](../papers/procedural-execution-llm-steps.md) is a useful warning shot for teams relying on multi-step prompt fidelity; accuracy falls hard as procedures lengthen. / [过程执行诊断](../papers/procedural-execution-llm-steps.md) 对依赖多步 prompt 可靠性的团队是个提醒：步骤一长，准确率掉得很快。
- [LightKV](../papers/lightkv-lvlm-kv-cache.md) points at a broader serving pattern: compress vision-token KV state without sacrificing general-purpose quality. / [LightKV](../papers/lightkv-lvlm-kv-cache.md) 指向一个更广的服务端模式：压缩视觉 token 的 KV 状态，同时尽量不牺牲通用能力。

## Patterns Emerging / 新出现的模式

- More of the AI stack is moving toward explicit lifecycle ownership: one root schema object, one authoritative prompt assembler, one active trace context. / AI 栈越来越倾向于显式的生命周期所有权：一个根 schema 对象、一个唯一的 prompt 组装器、一个活跃的 trace 上下文。
- The strongest agent papers this week are less about raw model cleverness and more about constraining execution, checking state, and preserving procedures. / 这一周最强的 agent 论文，重点不是模型有多聪明，而是如何约束执行、检查状态、保住步骤。
- KV cache work keeps splitting by modality and hardware generation, which suggests memory efficiency is still a live frontier rather than a solved baseline. / KV cache 的工作继续按模态和硬件代际分化，这说明内存效率仍然是活跃前沿，而不是已经解决的基础题。

## Risks Or Contradictions / 风险与矛盾

- Procedural accuracy can still look fine on short traces while collapsing on long ones, so benchmark scores alone can be misleading. / 过程执行在短 trace 上可能看起来不错，但到了长 trace 就崩，所以只看 benchmark 分数会误判。
- Tracing fixes often look tiny in diffstat but have outsized operational impact; these are easy to under-rank if you only skim code size. / tracing 修复在 diffstat 里常常很小，但运营影响很大；只看代码量很容易低估。
- Docs-only AI repo changes are usually low signal unless they expose a missing tool, workflow, or safety boundary. / 只改文档的 AI 仓库变更通常信号较低，除非它揭示了缺失的工具、流程或安全边界。
